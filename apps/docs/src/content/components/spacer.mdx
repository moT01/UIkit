`<Spacer>` is explicit layout gap. Reach for it when the surrounding
components should not carry margin of their own — typical in MDX
content where paragraphs and code blocks already own their spacing.

## Props

<PropTable
  rows={[
    {
      name: 'size',
      type: 'number',
      default: '2',
      description:
        '0–8 maps to `--space-0..--space-8` tokens. Any other number is raw pixels.'
    },
    {
      name: 'axis',
      type: "'vertical' | 'horizontal'",
      default: "'vertical'",
      description: 'Controls which dimension the gap consumes.'
    }
  ]}
/>

## Accessibility

Renders a presentation-only `<span>` with `aria-hidden="true"`. Screen
readers skip it entirely. Don't rely on it for semantic separation —
use a heading or landmark for that.

## Do & Don't

<DoDont>
  <Fragment slot='do'>
    <ul>
      <li>
        Prefer token steps over raw pixels — the spacing scale stays consistent.
      </li>
      <li>
        Use vertical spacers in MDX; horizontal spacers inside inline groups.
      </li>
      <li>Inline pixel values only for one-offs where no token fits.</li>
    </ul>
  </Fragment>
  <Fragment slot='dont'>
    <ul>
      <li>
        Don't add margin to neighbours when a spacer already exists — doubles
        gap.
      </li>
      <li>Don't nest spacers; use one with the combined step.</li>
      <li>
        Don't use as a spacer-between-list-items — reach for `gap` on the
        parent.
      </li>
    </ul>
  </Fragment>
</DoDont>
